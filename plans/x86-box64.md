# box64: per-program x86_64 emulation in existing arm64 distros

Run individual x86_64 Linux programs inside our existing aarch64 glibc
distros via [box64](https://github.com/ptitSeb/box64), ptitSeb's
x86_64→arm64 userspace translator. box64's signature design — translate
only the app's own code, redirect library calls to *native arm64* host
libraries via hand-written wrappers — means x86 programs would use our
existing production graphics path (arm64 libhybris EGL/GLES) with no
new graphics engineering.

**Status: plan, not started. Research-stage** — no box64 code has ever
run under tawcroot; every claim below about coexistence is unverified.

Sibling plans: [x86-fex-glibc.md](x86-fex-glibc.md) (FEX, same
in-distro shape), [x86-fex-bionic.md](x86-fex-bionic.md) (FEX with no
glibc layer). box64 and in-distro FEX are not mutually exclusive; the
exec-dispatch piece below is shared.

## Why box64 (vs FEX) — and why maybe both

- **Graphics for free (probably).** box64 wraps `libGL`/`libEGL`/
  `libGLESv2` etc. to native host libs by default; in our rootfs those
  are the libhybris wrappers that already work in production. FEX's
  default path emulates the whole guest GL stack instead. *(Verify:
  exact EGL/GLES wrapper coverage in current box64, and that wrapping
  tolerates hybris's non-Mesa libs.)*
- **16K-page tolerance.** box64 ≥0.2.8 claims 16K-page-host support;
  FEX hard-requires 4K-page kernels with no fix planned. All current
  tawc targets are 4K, but Android is pushing 16K, so box64 is the only
  hedge if a future device ships a 16K kernel.
- **Weaker whole-distro story.** box64 wants a native arm64 distro
  around it and mixes in native libs; it cannot host a self-contained
  x86_64 userland the way FEX's RootFS does. Complex/closed x86
  software with deep library assumptions is where box64's per-library
  wrapper model historically breaks down.
- **No 32-bit x86.** box64 delegates 32-bit to box86, which is an
  *armhf host binary* — a dead end for us regardless of box64: tawcroot
  `KILL_PROCESS`es 32-bit personality syscalls by design
  (notes/tawcroot/overview.md §"What it explicitly is not"). FEX does
  32-bit guests with 64-bit host syscalls. 32-bit x86 is out of scope
  for this plan.

## Pieces

### 1. tawcroot x86_64 exec dispatch (shared with FEX plan)

`exec_handler.c` currently classifies exec targets and returns a clean
`ENOEXEC` for wrong-arch ELFs (`img.e_machine != TAWC_EM_HOST`,
exec_handler.c:71). Add binfmt_misc-style dispatch: when the target is
an `EM_X86_64` (ET_EXEC/ET_DYN) ELF and an emulator is configured,
rewrite the exec to `<emulator> <orig-path> <orig-argv[1..]>` the same
way shebangs are resolved today. Configuration via a new tawcroot CLI
arg (e.g. `--x86-emulator=/usr/local/bin/box64`), plumbed from the
Kotlin spawn path like binds are.

Design points to settle during implementation:
- argv[0]/`AT_EXECFN` fidelity for the emulated program (box64 and FEX
  both have conventions for receiving the real path; check what each
  needs).
- Shebang chains ending in an x86_64 interpreter should dispatch too
  (classification already resolves the chain, exec_handler.c:99-107).
- Recursion guard: the emulator binary itself must exec as a normal
  arm64 guest.
- Per-distro vs per-spawn configuration; probably per-distro metadata.

This piece is emulator-agnostic and is the single tawcroot change both
this plan and x86-fex-glibc.md need. Land it once, first.

### 2. box64 binary into the rootfs

box64 is a single arm64 binary (plus wrapper data). Packaging options,
decide later:
- Build from source in `deps/` via a script (like libhybris) and pack
  as an asset installed into the rootfs — most control, our default
  pattern.
- Distro packages where they exist (AUR `box64`, Debian has no official
  package as of last check — *verify current packaging*).

box64 is a CMake/C project with a glibc target; cross-compiling
arm64-glibc from our host toolchain or building inside a rootfs both
plausible. Decide later.

### 3. x86_64 library tree for unwrapped libs

box64 only wraps common libraries; everything else needs real x86_64
`.so`s on `BOX64_LD_LIBRARY_PATH`. Minimum viable: an x86_64 glibc +
common-libs tree in the rootfs (e.g. extracted from our own x86_64
distro bootstraps — the installer already knows how to fetch Arch/
Debian/Void x86_64 rootfses for the emulator). How much tree, where it
lives, and whether it's installed on demand: decide later.

### 4. Bring-up workloads

Suggested ladder: static x86_64 hello-world → dynamic hello-world →
x86_64 CLI tool doing real syscall traffic (`git`, `python`) → an SHM
Wayland client → an EGL/GLES Wayland client through wrapped libs.

## Open questions / risks

- **box64 JIT under tawcroot is completely untested.** Known Android
  friction from the field: address-space width misdetection on some
  SoCs (box64 #2239), bionic-related syscall-return truncation in
  Termux glibc setups (should not apply — we're glibc all the way —
  but the class of bug exists). tawcroot adds its own wrinkles: fixed
  non-PIE base at `0x2000000000`, SIGSYS ownership, guest-seccomp
  denial. Any of these could interact with box64's dynarec; needs a
  device probe before believing this plan.
- **Signal coexistence.** box64 uses SIGSEGV/SIGILL handlers
  internally; tawcroot shadows guest `SIGSYS` only. Presumed fine,
  unverified.
- **Wrapper/hybris mismatch.** box64's GL wrappers assume Mesa-ish
  native libs; libhybris's EGL/GLES are unusual (Android blobs behind
  a glibc shim). If wrapping trips on them, the fallback is
  `BOX64_LIBGL` pointing at x86_64 Mesa llvmpipe (software).
- **W^X / execmem**: Android grants `execmem` to app domains (JITs
  work in Winlator), so anonymous RWX should be available. Verify
  under our exact SELinux context early.
- **Wrapped-vs-emulated libc boundary bugs** are box64's known
  long-tail (threading, dlopen corner cases). Expect per-app
  whack-a-mole; that's inherent to the wrapping model.

## Testing

- tawcroot handler/integration tests for the exec-dispatch piece
  (dispatch fires for EM_X86_64, recursion guard, ENOEXEC preserved
  when no emulator configured) — these don't need box64 itself; a stub
  "emulator" script suffices.
- Device integration test running an x86_64 hello-world through the
  full stack once packaging exists.

## Out of scope

- 32-bit x86 (see above; FEX territory if ever needed).
- Whole-x86_64-distro installs — that's the FEX plans.
- Wine/Windows programs.
